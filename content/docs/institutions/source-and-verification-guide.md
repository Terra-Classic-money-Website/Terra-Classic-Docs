Institutional readers should use Terra Classic Docs as a source-aware explanation layer, not as a live analytics terminal.

The docs should clarify product status, risks, and interpretation. Live values should be verified through external data products, explorers, wallets, governance interfaces, market venues, or direct chain queries.

## What the docs can verify

The docs can help verify:

- whether a product is described as live, proposed, governance-accepted, or historical
- what source material a page is based on
- which risks should be considered before integration
- whether a proposed feature is being presented cautiously
- where a reader should go for live evidence

## What external tools should verify

External tools should verify:

- balances
- transactions
- blocks
- governance vote state
- validator data
- prices
- liquidity
- market depth
- live endpoint behavior
- production wallet support

Do not treat static docs pages as a substitute for current operational data.

## Due diligence rule

For any Terra Classic integration, separate four questions:

| Question | Where to look |
| --- | --- |
| What does the concept mean? | Docs pages. |
| Is it live or proposed? | Docs status language plus current governance/release evidence. |
| What are the current values? | External explorers, dashboards, market tools, or direct queries. |
| What are the risks? | Docs risk pages plus institution-specific review. |
