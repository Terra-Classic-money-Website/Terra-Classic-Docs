A proposal should earn trust before it earns votes.

The goal is not to make every voter a developer, lawyer, or economist. The goal is to give users a simple framework for spotting serious proposals and rejecting weak ones.

## First question

Ask one question before anything else:

> If this proposal passes, what exactly changes?

If the answer is unclear, the proposal is not ready.

## Proposal quality checklist

A serious proposal should make the following clear:

| Question | Why it matters |
| --- | --- |
| What problem does it solve? | Governance should not approve vague activity. |
| What exact action happens? | Voters need to know whether this is code, funding, signaling, or coordination. |
| Who is responsible? | "The community" is not an execution owner. |
| What budget is requested? | Funding requires scope, amount, custody, and deliverables. |
| What evidence supports the claim? | Good proposals show sources, data, code, or comparable examples. |
| What are the risks? | Serious teams disclose tradeoffs before opponents do it for them. |
| What is the timeline? | Open-ended promises are hard to evaluate. |
| How is success measured? | Voters need proof after the vote. |
| What happens if execution fails? | Governance needs an accountability path. |

Weak proposals ask governance for trust without giving governance enough information to make a decision.

## Identify proposal type

Not all proposals carry the same risk.

| Proposal type | Main review focus |
| --- | --- |
| Text or signaling | Is the direction clear, useful, and honestly framed? |
| Community pool spend | Is the budget justified and milestone-based? |
| Software upgrade | Are code, testing, release notes, and upgrade coordination credible? |
| Parameter change | What second-order effects does the change create? |
| Product proposal | Is it live, proposed, funded, audited, or only accepted as a direction? |
| Stablecoin or market mechanism | Are collateral, oracle, liquidity, legal, and user-safety risks addressed? |

## Check the execution path

A proposal can pass and still fail operationally.

Before voting `Yes`, check whether the execution path includes:

- named owner
- delivery milestones
- public repository or artifact
- acceptance criteria
- update cadence
- testing or review plan
- budget release logic
- final delivery proof

If a proposal asks for money but does not define acceptance criteria, voters should be skeptical.

## Check source quality

Prefer proposals that link to:

- code repositories
- design documents
- technical specifications
- governance discussions
- dashboards or queryable data
- audit reports or review notes
- previous delivery history
- clear wallet or multisig addresses where relevant

Screenshots and social posts can support a proposal, but they should not be the only evidence for a major decision.

## Red flags

Be careful with proposals that:

- promise price outcomes
- use urgency to avoid review
- hide who controls funds
- confuse passed governance with completed implementation
- claim official status without proof
- omit risks
- request funding without milestones
- rely on vague "community will decide later" language
- attack critics instead of answering technical or economic questions
- turn a high-risk product into a branding exercise

## Product proposals need extra discipline

Terra Classic has several L1 product directions, including [Swap Protocol](/swap-protocol/overview/) and [Forex Protocol](/forex-protocol/overview/).

For product proposals, voters should separate:

- governance acceptance
- implementation funding
- code completion
- testing
- audits
- liquidity
- wallet support
- mainnet availability

Those are different stages. A proposal passing does not automatically make a product live.

## Investor-grade standard

For major proposals, use this standard:

> Claim, artifact, verifier, timestamp.

A proposal should say what is being claimed, show the artifact that proves it, make it possible for others to verify, and include a timestamp or checked date.

If a proposal cannot be verified, treat it as commentary until evidence exists.
