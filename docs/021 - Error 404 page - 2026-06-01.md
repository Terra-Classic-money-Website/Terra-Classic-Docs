# Error 404 page

Created: 2026-06-01

## Context

The docs app previously resolved unknown routes to the first docs page. That made broken or moved URLs look like the documentation start page, which is poor wayfinding and creates SEO ambiguity for GitHub Pages fallback traffic.

## Decision

Add a dedicated 404 state inside the existing docs shell rather than creating a standalone marketing-style page or a markdown docs article.

The 404 page should:

- keep the user inside the Terra Classic Docs interface
- avoid promotional copy
- route users back to the docs start page or the main Terra Classic website
- preserve the existing sidebar search as the primary recovery path
- avoid highlighting an unrelated active navigation item

## Implementation

- `src/App.tsx` now treats an unknown path as `null` instead of falling back to the start page.
- `src/components/docs/DocsShell.tsx` accepts a missing active page and hides breadcrumbs/table-of-contents for the shell while rendering the 404 page's own breadcrumbs.
- `src/styles/docs.css` defines the reusable 404/error action pattern.
- `src/designsystem.tsx` includes a 404 Error State sample so future error or empty states follow the same visual language.
- `scripts/write-static-routes.mjs` writes `dist/404.html` with page-not-found metadata and `noindex` robots metadata.
- `scripts/validate-static-seo.mjs` fails the build if `404.html` loses the expected title, canonical URL, or `noindex` metadata.

## Validation

Full handoff gate:

```bash
npm run check
```

Result: passed.

Rendered QA:

- Local URL: `http://127.0.0.1:5174/not-a-real-page/`
- Browser path: in-app browser runtime was attempted first, but this session's browser URL policy blocked the local target, so standalone Playwright was used for rendered validation.
- Desktop viewport: `1440x1000`
- Tablet viewport: `900x900`
- Mobile viewport: `390x844`

Result: the unknown route rendered the 404 page title and heading in all checked viewports, the primary action returned to the docs start page, and Playwright reported no console warnings, console errors, page errors, or framework overlays.
